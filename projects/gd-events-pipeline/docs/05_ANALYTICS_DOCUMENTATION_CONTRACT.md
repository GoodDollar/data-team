# Analytics Documentation Contract

This contract defines the minimum documentation, testing, and AI-readiness requirements for every new GoodDollar onchain analytics source, dbt model, metric, and AI-exposed dataset.

It exists for one reason: self-service analytics must be concrete, reproducible, traceable to BigQuery, and grounded in actual onchain data. A fast answer that cannot be trusted is a liability.

## Scope

This applies to all future additions:

- new chains
- new contracts
- new contract events
- new raw BigQuery source tables
- new dbt staging models
- new semantic entities
- new marts
- new metrics
- new dashboard or AI self-service surfaces

Nothing is considered AI-ready until it satisfies this contract.

## Principles

1. **L1/raw is the audit substrate.** The AI analyst may use raw sources for proof, reconciliation, and investigations, but only after loading the relevant dbt docs, lineage, and business definitions.
2. **Business meaning lives above raw.** Raw sources preserve what happened onchain. Staging cleans and normalizes. Semantic models define business entities. Marts serve approved dashboard shapes.
3. **Broad access, strict protocol.** A future AI analyst can traverse L1/L2/L3, but every answer must cite tables/models, SQL, freshness, and caveats.
4. **Latest closed day is the default.** Business-facing analytics must exclude the current partial day unless explicitly documented as partial/intraday.
5. **Ambiguity is a handled state.** If a user asks for an ambiguous term such as "users", "transactions", or "activity", the system must ask targeted disambiguation questions before running analysis.
6. **Docs are part of the data product.** A model with unclear grain, units, lineage, or business rules is not finished.

## Required Contract Registry

Every new contract or event family must have a registry entry before production backfill.

Required fields:

| Field | Requirement |
|---|---|
| Domain | Business area, such as claims, invites, token transfers, identity, staking, governance |
| Contract name | Human name and onchain contract name |
| Chain/network | XDC, CELO, Ethereum, Fuse, or other |
| Chain ID | Numeric chain ID |
| Contract address | Lowercase `0x` address, one row per chain deployment |
| Deployment/from block | First block to backfill from, with source/evidence |
| ABI/source path | Path to ABI or verified contract reference |
| Events included | Event names, signatures, and why each event matters |
| Events excluded | Known events intentionally not ingested and why |
| Finality policy | Confirmation/reorg rule for the chain |
| Ingestion owner | Pipeline code owner or operator |
| Analytics owner | Person responsible for business definitions |

## L0/L1 Raw Source Requirements

Each raw source table declared in dbt must document:

| Item | Requirement |
|---|---|
| BigQuery relation | Project, dataset, table |
| Source owner | Pipeline-written, dbt-read-only |
| Grain | Exactly what one row represents |
| Natural key | Usually `(network, tx_hash, log_index)` |
| Dedup strategy | How duplicates are prevented and detected |
| Partition/cluster | BigQuery partition and clustering fields |
| Common columns | Network, chain ID, block fields, tx fields, log fields, contract address, event name, ingested_at |
| Event-specific columns | One row per event field, with type and nullability |
| Units/decimals | Raw units, token decimals, and conversion rule |
| Sentinel values | Zero address, contract address, nulls, or other special values |
| Accepted event names | dbt accepted-values test where possible |
| Raw preservation rule | What is preserved exactly from chain versus generated by pipeline |
| Source freshness | Warning/error thresholds and loaded_at field |
| Known caveats | Reorg exposure, missing event types, API limits, nullable fields |

Required tests where applicable:

- not null on `network`, `chain_id`, `block_number`, `block_timestamp`, `tx_hash`, `log_index`, `contract_address`, `event_name`, `ingested_at`
- unique natural key, usually `(network, tx_hash, log_index)`
- accepted values for `network`, `chain_id`, and `event_name`
- freshness checks on `ingested_at`
- row-count or block-range reconciliation after backfills

## Staging Model Requirements

Every raw source table gets exactly one staging model before semantic models consume it.

Staging models must document:

| Item | Requirement |
|---|---|
| Source table | The exact `source()` relation |
| Grain | Same row grain as raw source |
| Transform scope | Minimal cleaning only: lowercase addresses, type casting, renames |
| Business logic | Explicitly absent; business logic belongs in Semantic |
| Closed-day rule | Must filter to `latest_closed_date()` unless explicitly intraday |
| Column mapping | Raw column to staged column, including renames |
| Type conversions | Raw type, staged type, failure behavior such as `SAFE_CAST` |
| Tests | not-null, uniqueness, accepted values, and relationships where applicable |

Staging must not classify business concepts, join domains, or aggregate metrics.

## Semantic Model Requirements

Semantic models are the canonical business definitions.

Each semantic model must document:

| Item | Requirement |
|---|---|
| Business purpose | What question or entity this model exists to answer |
| Grain | One row per what |
| Sources | Upstream dbt refs and the reason each is used |
| Inclusion scope | Which records are included and excluded |
| Business rules | Every CASE, filter, eligibility rule, classification, and join rule in plain English |
| Units | Human-readable units and conversion from raw values |
| Time semantics | Which timestamp drives analysis and why |
| Network semantics | Whether logic is chain-local or cross-chain |
| Edge cases | Duplicates, sentinel values, missing data, timing anomalies |
| Downstream use | Which marts/dashboards/AI questions should use it |
| When not to use | Known cases where another model is safer |

Required tests where applicable:

- unique primary grain key
- not-null dimensions required for analysis
- accepted values for classifications
- relationships to upstream entities
- reconciliation from source counts to semantic counts
- custom tests for business invariants

## Mart Requirements

Marts are approved consumer surfaces for dashboards, BI tools, and first-pass AI answers.

Each mart must document:

| Item | Requirement |
|---|---|
| User-facing purpose | What stakeholder question this mart answers |
| Grain | One row per date/network/entity/stage/etc. |
| Refresh cadence | Full rebuild, incremental, snapshot, or other |
| Data freshness | Latest closed day or explicit partial/intraday exception |
| Source models | Upstream Semantic refs |
| Metrics | Every measure and formula |
| Dimensions | Every supported slice/filter |
| Cumulative logic | Window definitions and reset partitions |
| Approximation warning | Any approximate metric must be labeled in the column name and docs |
| Known caveats | Timing anomalies, non-monotonic funnels, or source limitations |
| Expected consumers | Looker Studio, Metabase, AI analyst, exports |

Required tests where applicable:

- unique combination of grain columns
- not-null grain columns
- non-negative counts/amounts
- accepted values for dimensions
- reconciliation to upstream Semantic models
- no current partial day unless explicitly intraday

## Metric Contract Requirements

Every metric exposed to dashboards or AI must have a glossary entry.

Required fields:

| Field | Requirement |
|---|---|
| Metric name | Canonical name users and AI should use |
| Plain-English definition | Business meaning in one or two sentences |
| Formula | SQL or dbt metric formula |
| Source model | Canonical model/table |
| Grain | The level at which the metric is valid |
| Time field | Timestamp/date used for filtering |
| Default freshness | Latest closed day unless explicitly partial |
| Dimensions | Supported slices |
| Synonyms | Words users might say that map to this metric |
| Forbidden meanings | Similar terms that must not be conflated |
| Example questions | Questions this metric can answer |
| Disambiguation prompt | What the AI should ask if intent is unclear |
| Owner | Person/team accountable for definition changes |

## AI Exposure Requirements

A model or metric can be exposed to Metabase AI or a custom GoodDollar AI analyst only when:

- dbt docs include model and column descriptions
- grain is explicit
- upstream lineage is clear from dbt refs/sources
- key columns have tests
- accepted values are documented and tested where possible
- business glossary terms exist for exposed metrics/entities
- freshness behavior is documented
- known caveats are documented
- at least one golden question validates the expected answer

For AI analyst answers, required output metadata:

- interpreted question
- assumptions made
- disambiguation answers received, if any
- tables/models used and why
- SQL executed
- BigQuery job ID or equivalent execution reference
- data freshness / latest closed date
- row counts or reconciliation checks where relevant
- caveats and known limitations
- chart/table result, if requested

## Definition Of Done

A new source/model/metric is production-ready when:

1. It is documented according to its layer requirements.
2. It has dbt tests appropriate to its grain and risk.
3. It compiles and runs in dbt.
4. It has passed row-count/KPI reconciliation against upstream data where possible.
5. It obeys the latest-closed-day contract unless explicitly documented otherwise.
6. It has glossary coverage for any user-facing terms.
7. It has at least one verified golden question if exposed to self-service AI.
